Product Requirements · 살아있는 단일 SoT

productune-lite

v1.1 매 버전 이 파일을 제자리에서 다듬는다 — git 이 버전 히스토리. 각 stage 품질 규율을 full productune 에서 4개 이식하되, 전부 PO 판단 트리거로.

이 제품은 무엇인가

productune-lite — 코드를 모르는 기획자를 위한, PO 판단으로 굴러가는 가벼운 제품 워크플로우. v1.0 출시됨. 이 레포가 곧 제품이며, "제품 변경" = productune-lite 의 doctrine 마크다운(페르소나별 habit / bookshelf 규칙 파일) 편집이다.

lite 계약 — 절대 위반 금지

ticket 파일 없음 · phase-close gate 없음 · enforcement hook 없음 · migration 프레임워크 없음. 이 모든 것을 대체하는 단 하나 = PO 의 판단. 라이프사이클: Define → Build → Ship + idle.

누구를 위한 것인가

  • 1차 사용자 = 기획자(비개발자). WHAT 은 알지만 HOW(코드)는 모른다. PO(pdtl-po)와 대화만 하고, PO 가 designer / developer / qa 를 오케스트레이션한다.
  • 간접 사용자 = 워커 페르소나들(designer / developer / qa). doctrine 를 읽고 행동한다. v1.1 변경분의 실제 "독자"는 이들의 habit / bookshelf 파일이다.

핵심 잡 (jobs to be done)

  • 아이디어 한 줄에서 명확한 PRD 로 수렴한다 — Define
  • PRD 를 작동하는 제품으로 만든다 — user-facing 이면 디자인부터 — Build
  • 실제로 배포·검증하고 배운 것을 기억한다 — Ship

v1.0 이 이 뼈대를 이미 굴린다. v1.1 은 각 stage 의 품질 규율을 4개 이식한다 — 단, 전부 hook/gate/ticket 이 아니라 PO 판단 트리거 + doctrine 실천으로.

v1.1 범위 — 이식할 4개 역량

공통 원칙: 각 역량은 full 의 가치만 가져오고 강제 기계장치는 버린다. 산출물은 항상 "PO/워커가 이제 무엇을 다르게 하는가"로 정의한다. 역량 3·4(준수+보안)는 Build→Ship readiness pass 하나로 묶어 서술한다.

01

PRD clarity loop — 점수 rubric 기반 명확성 루프

Define

무엇을 이식

구조화된 clarity-loop 규율(빠짐없는 커버리지 + 가장 안 명확한 곳부터 한 번에 하나씩) + 명확성 점수 A = 1 − Σ(clarityᵢ × weightᵢ)(ready A ≤ 0.05, weight 표, 차원별 clarity 점수)를 되살려 이식. 단 성격은 hook/gate 가 아닌 판단 rubric — 감 대신 점수로 더 잘 판단. 5-iter 는 hard cap 아닌 soft 가이드.

점수 rubric + 커버리지 (weight 기본값)

문제 & 대상 사용자0.18
핵심 잡(JTBD) / 원하는 결과0.14
범위 경계 (in / out / later)0.13
수용 조건 ("done")0.12
위험 & 가정0.10
성공 신호 (north star + input)0.09
솔루션 형태 (가설)0.08
외부 의존 / 통합0.06
브랜드 / UX 방향0.05
운영 / GTM / 런칭0.05

designer 가 차원별 clarity(0–1)를 매겨 A 계산. A ≤ 0.05 → ready, 아니면 가장 낮은 clarity × 높은 weight 차원을 골라 단일 needs_info 질문(PO 릴레이). weight 는 기본값 — designer 가 프로젝트에 맞게 조정 가능.

no-hook / no-gate 준수

lite 는 hook 이 없으니 점수는 무엇도 강제 안 함 — A ≤ 0.05·5-iter 자동전환/차단 없음. designer 계산·PO 읽고 판단·PO finalize 언제든 override.

02

디자인 시스템 → 하이파이 (순차, 사용자 승인 후)

Build · design

무엇을 이식

full P2 체인 S1(DS 제안) + S5(하이파이)의 가치 — 토큰 잡힌 DS 를 먼저 보여 승인받고 그 위에 하이파이를 얹는다. 순서와 3안 뽑는 기준만 가져오고, 5-스텝 gated 체인 · branch A/B/C · design ticket 같은 기계장치는 버린다.

순서 — 반드시 순차

① DS 목업 렌더·제시② 사용자 승인·선택③ 하이파이 + docs/design.md

승인 전엔 하이파이로 안 간다. ② 가 lite 의 load-bearing fork 확인 지점.

① 단계 — 사용자 방향 분기

  • 방향 있으면 ("이 디자인으로" / 브랜드 가이드) → 3안 생략, 그 방향으로 DS 1안만 세워 제시
  • 딱히 없으면 → DS 3안 렌더해 제시, 사용자가 고름

3안 기준 — productune S1 규율 반영

  • 3-mix — A·B = style-library/index.md Fit anchor 적응(클론 아님), C = 웹서치 기반 진짜 새 방향(발산)
  • mood brief 먼저 — 표면 유형 · 오디언스 온도 · mood 형용사 3–5 · 브랜드 제약 도출 후 anchor 선정
  • famous-brand cap ≤ 1 — top-tier 디폴트(linear/stripe/vercel/claude/notion/airbnb 급) 최대 1개
  • divergence rule — 어느 둘이든 4개 mood label 중 ≥ 2 상이 + 폰트·셰이프·레이아웃 차이("색만 다른 같은 안" 금지)
  • 렌더 요건 — 지정 폰트 실제 로드 · anti-default 패스 · 게이트 전 render-verify

브랜드 에셋(logo/favicon/og)은 확정 DS 파생 — 이미지 생성 필요 시 external_tool_recommendation 로 핸드오프. 거부는 정규 판단 루프(재-dispatch, gated 스텝 아님).

no-hook / no-gate 준수

DS→하이파이 승인은 판단 컨펌이지 hook 게이트가 아니다. S1–S5 스텝 게이트 · branch 분류 · design ticket 없음. productune 은 3안 기준만 참조 — 기계장치는 안 가져온다.

03·04

Build → Ship readiness pass (준수 점검 + 보안 점검)

Build → Ship

묶는 이유 · 성격

역량3(DS 준수) + 역량4(보안)를 하나의 lifecycle 순간에 묶는다 — Build→Ship 전환 시 PO 가 표준으로 돌리는 pass. hook 게이트와 순수 즉흥판단 사이의 중간: 아무것도 안 막지만(N/A 는 memory.md 한 줄로 스킵) stage 경계에 묶여 안 까먹는 soft ritual.

① DS 준수 점검 — user-facing 이면

PO 가 designer 또는 QA(어디 실을지 PO 판단)에 dispatch. DS 일관성 · 타이포 · 컬러 · 스페이싱 · 에셋(logo/favicon/og/meta) · 하이파이 일치(없으면 이 항목만 N/A) · anti-default 미학. 문제 발견 → patch 루프, 봐준 항목은 memory.md 한 줄.

② 보안 점검 — 표면 조건부 세트 (QA)

  1. secrets (대개 공통) — 하드코딩 크리덴셜·키·토큰 없음, 런타임 주입
  2. deps (패키지 매니저 표면) — audit high/critical 미해결 없음, 락파일 정합
  3. data-exposure (대개 공통) — PII 과수집·로그·전송 없음
  4. dist-integrity (배포 산출물 표면) — 정규 빌드 재현, 가능 시 서명/체크섬
  5. entry-surface (CSP·헤더) (웹 진입 표면) — unsafe-* 과도 개방 아님
  6. platform-hardening (데스크톱/Electron 등) — contextIsolation·IPC 검증 등

표면 적용/N/A 는 PO 가 매번 판단(순수 CLI → 5·6 N/A 등). 6개는 완결이 아닌 출발 세트. 문제 발견 → Ship-내부 패치 루프(stage:"ship" 유지).

no-hook / no-gate 준수

readiness pass 는 stage 경계에 묶인 soft ritual — hook 아님, no-waiver gate 아님. "언제 도나 · 무엇이 N/A · 무엇을 봐주나"는 전부 PO 판단, ticket 없이 dispatch + return envelope.

non-goals — v1.1 이 하지 않는 것

  • ticket 파일 / phase-close gate / enforcement hook / migration 프레임워크 신설 — 계약 위반
  • 5-스텝 gated 디자인 체인(S1–S5), branch A/B/C 분류, design ticket 발행
  • no-waiver mandatory 게이트, tier-memory 기계장치 — readiness pass 는 soft ritual, 점수는 판단 rubric(둘 다 자동 차단/전환 아님)
  • 강제 iter hard cap — 5-iter 는 soft 가이드로만 (점수 수학 자체는 v1.1 에서 도입)
  • lite 를 heavy productune 으로 만들기 — lite 는 lite 로 남는다

v1.1 "done" 의 모습

이식 4개가 각각 doctrine 마크다운 편집으로 반영되고, 실행 시 아래가 관찰된다:

  1. Clarity loop (점수 rubric) — doctrine/designer/habit.md(PRD 섹션)에 명확성 점수 A=1−Σ(clarity×weight) + weight 표 + 커버리지 차원이 판단 rubric으로. designer 가 차원별 clarity 매겨 A 계산·가장 약한 곳부터 단일 질문, PO 는 A(ready ≤ 0.05) 읽어 판단하되 언제든 finalize override. 5-iter 는 soft 가이드.
  2. DS → 하이파이 순차 — doctrine/designer/habit.md + lifecycle.md(Build)에 "① DS 제시 → ② 승인 → ③ 하이파이" 순차와 방향 유무별 1안/3안 분기 명문화. 3안은 mood brief → Fit anchor(A·B)+웹서치(C)+famous-cap+divergence rule 규율.
  3. Ship 진입 readiness pass — doctrine/designer/habit.md + doctrine/qa/habit.md + lifecycle.md(Build→Ship)에 PO 표준 pass 명문화: user-facing 이면 ① DS 준수 점검(designer/QA, drift → patch) + ② 표면 조건부 보안 점검(QA, 이슈 → Ship-내부 패치). stage 경계에 묶인 soft ritual — N/A 는 memory 한 줄, 아무것도 강제로 안 막음.

전 항목이 hook/gate/ticket 없이 순수 PO 판단 + doctrine 실천으로 표현됐음이 각 편집에서 확인된다. lite 의 3-stage 정체성과 "판단이 강제를 대체한다"는 계약이 v1.1 후에도 성립하면 done.

확정 사항 · 가정

  • 확정 — lite 로 만드는 제품의 표면은 다양하다(웹·데스크톱·CLI 등). → 역량4 는 표면 조건부 세트, PO 가 매번 적용/N/A 판단.
  • 확정 — 역량3 준수 점검 주체는 PO 가 Build 시 designer 또는 QA 로 판단해 실음(고정 안 함).
  • 가정 — conformance/security 의 "무엇을 봐주나"는 PO 판단 + docs/memory.md 한 줄. waiver 필드/기계장치 신설 안 함.
  • 가정 — PRD 는 사용자가 직접 읽으므로 고정 템플릿 없이 한국어로, versions/ 는 건드리지 않음.
